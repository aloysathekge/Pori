# Aloy Event Surface — typed composition specification

_Version 1.0, 2026-07-15. This specification defines the Event Surface named
in [`aloy-vision.md`](./aloy-vision.md). It replaces the assumption that a
Surface is only a fixed Event sidebar._

## 1. Decision

An **Event Surface is a durable, live workspace composed from allowlisted,
typed blocks and rendered by trusted React components**.

It is not:

- the webapp, desktop app, or website as a whole;
- a screenshot or one fixed project template;
- arbitrary HTML, React, CSS, JavaScript, or an iframe generated by a model;
- a second source of truth competing with Tasks, Proposals, receipts, files,
  Runs, or Trail;
- a chat response dressed up as application state.

The existing Tasks, Decisions, Files, Notes, and Trail pane is the first static
projection. The target system generalizes it so an Event such as **Trip to San
Francisco** can contain a map and itinerary, while **Career OS** can contain a
company comparison and cited research document.

## 2. Workspace behavior

Conversation and Event Surface are peers inside the Event workspace:

```text
Event workspace
├── continuous Conversation
└── Event Surface canvas
    ├── system blocks
    └── domain blocks
```

The user controls three presentation modes:

1. **Conversation focus** — the Surface is collapsed to context and attention;
2. **Split** — Conversation and Surface are both visible and resizable;
3. **Surface focus** — the canvas receives the main workspace width.

A map, comparison, or document may therefore use the main view rather than
being squeezed into a narrow sidebar. Aloy may update data and badges live, but
must not steal focus, change the selected block, or rearrange the layout while
the user is interacting. On narrow screens the two peers become explicit tabs
or full-screen sheets.

## 3. Trust boundary

The backend owns the Surface document and validates every block. The frontend
owns the renderer registry. Aloy may select and populate allowed block types;
it never supplies executable UI code.

```text
durable Event records ─┐
typed working blocks ──┼─> Surface projector ─> SurfaceDocument JSON
user layout prefs ─────┘                              │
                                                     ▼
                                      trusted React renderer registry
```

Canonical records remain authoritative:

- Task state comes from `Task`;
- execution state comes from `Run` and checkpoints;
- decisions come from `ActionProposal`;
- committed consequences require receipts;
- files and artifacts come from `StoredFile`;
- narrative history comes from `EventTrailEntry`;
- domain research may live in a validated Surface block and linked artifact,
  with evidence references.

A Surface block may project those records, reference them, or contain bounded
working data. It may not turn an unverified claim into committed reality.

## 4. Surface document contract

The server returns one versioned document per Event:

```json
{
  "event_id": "evt_123",
  "schema_version": 1,
  "revision": 42,
  "generated_at": "2026-07-15T12:00:00Z",
  "layout": {
    "mode": "split",
    "columns": 12,
    "placements": []
  },
  "blocks": [],
  "stale": false
}
```

`revision` changes monotonically when semantic Surface data changes. SSE sends
an invalidation carrying the Event and latest revision; reconnect uses durable
cursors and refetches any missed terminal state. User-only view preferences
such as selected block or split ratio do not create Trail noise.

Every block has a common envelope:

```json
{
  "id": "block_trip_map",
  "type": "map",
  "version": 1,
  "title": "San Francisco plan",
  "posture": "working",
  "layout_hint": { "w": 8, "h": 7 },
  "data": {},
  "source_refs": [],
  "created_by": { "kind": "run", "id": "run_123" },
  "last_run_id": "run_123",
  "updated_at": "2026-07-15T12:00:00Z"
}
```

Common rules:

- `type` and `version` must exist in the server and client registries;
- `data` is validated against the block's schema before persistence;
- identifiers referenced by a block must belong to the same tenant and Event;
- text, arrays, URLs, coordinates, and nested payloads have explicit bounds;
- source references carry URL/title/retrieval time or durable record IDs;
- agent writes cannot overwrite or delete a user-authored block unless the user
  explicitly transfers management or performs the action directly;
- unknown block types render an unsupported-block state, never execute code;
- destructive or external controls create or resolve Proposals through normal
  APIs rather than embedding provider calls in a renderer.

## 5. V1 block registry

### 5.1 System blocks

These are projections over canonical product records:

| type | purpose | authoritative source |
|---|---|---|
| `status` | Event goal, phase, and current summary | Event |
| `task_list` | executable work and controls | Task + Run |
| `decision_list` | pending/resolved consequences | ActionProposal + receipt |
| `file_list` | uploads and produced artifacts | StoredFile |
| `trail` | pageable semantic narrative | EventTrailEntry |
| `notes` | user-maintained Event notes | Event metadata or typed note record |

### 5.2 Domain blocks

These make the Surface useful beyond one template:

| type | examples | required posture/evidence behavior |
|---|---|---|
| `document` | cited report, plan, brief | working content; sources attached |
| `table` | company comparison, options matrix | column schema; per-row sources |
| `card_collection` | companies, hotels, candidates | stable item IDs; sources per item |
| `timeline` | itinerary, launch plan, application stages | ordered dated/undated items |
| `map` | trip locations, route candidates | bounded coordinates; sourced places |

Charts, calendars, kanban, media galleries, and third-party plugin blocks may
be added through versioned registry changes after a real use case requires
them. They are not generic arbitrary payload renderers.

The `map` wire contract is provider-neutral. A React renderer may use MapLibre,
Google Maps, Mapbox, or another reviewed adapter, but provider configuration is
not stored in the block and must not expose credentials. The implementation
phase must select and document the tile/geocoding provider, privacy posture,
attribution requirements, offline behavior, and failure fallback before the
renderer ships.

## 6. Example: Trip to San Francisco

```json
{
  "event_id": "evt_sf",
  "schema_version": 1,
  "revision": 9,
  "layout": { "mode": "surface_focus", "columns": 12 },
  "blocks": [
    {
      "id": "block_sf_map",
      "type": "map",
      "version": 1,
      "title": "San Francisco",
      "posture": "working",
      "layout_hint": { "w": 8, "h": 8 },
      "data": {
        "center": { "latitude": 37.7749, "longitude": -122.4194 },
        "markers": [
          {
            "id": "place_sfo",
            "label": "SFO",
            "latitude": 37.6213,
            "longitude": -122.379,
            "source_ref": "source_sfo"
          }
        ]
      },
      "source_refs": [
        {
          "id": "source_sfo",
          "url": "https://www.flysfo.com/",
          "title": "San Francisco International Airport",
          "retrieved_at": "2026-07-15T11:00:00Z"
        }
      ],
      "created_by": { "kind": "run", "id": "run_trip_research" },
      "last_run_id": "run_trip_research"
    },
    {
      "id": "block_sf_itinerary",
      "type": "timeline",
      "version": 1,
      "title": "Draft itinerary",
      "posture": "working",
      "layout_hint": { "w": 4, "h": 8 },
      "data": { "items": [] },
      "source_refs": [],
      "created_by": { "kind": "user", "id": "user_123" },
      "last_run_id": null
    }
  ],
  "stale": false
}
```

Research may add or revise markers and itinerary items. Booking a flight or
hotel remains a Proposal. A marker is not a booking, and a draft itinerary is
not committed external reality.

## 7. Construction and mutation

Surface construction has two paths:

1. **System projection:** the backend deterministically supplies required
   system blocks from canonical rows.
2. **Typed composition:** the user or Aloy creates/updates/removes domain blocks
   through a validated Surface service and tool.

The agent-facing tool accepts intent such as:

```text
surface_upsert_block(event_id, block_id, expected_revision, idempotency_key,
                     type, version, title, data, source_refs)
```

The tool resolves tenancy from the authenticated Run; the model does not choose
organization or user IDs. Successful semantic mutations append
`surface_block_created`, `surface_block_updated`, or `surface_block_removed`
Trail entries with bounded before/after summaries. Repeated identical writes
are idempotent and do not create duplicate revisions or Trail entries.

Block mutation, Event Surface revision, and semantic Trail append commit in one
transaction. `expected_revision` rejects stale concurrent edits instead of
silently losing work. Agent-managed blocks retain their creating and last
writing Run; user-managed blocks are protected from autonomous overwrite or
removal.

User layout changes use a separate endpoint and preference record. Aloy may
recommend a layout, but V1 does not rearrange a user's canvas automatically.

## 8. React rendering contract

The app owns an explicit registry:

```tsx
const surfaceRenderers = {
  status: StatusBlock,
  task_list: TaskListBlock,
  decision_list: DecisionListBlock,
  file_list: FileListBlock,
  trail: TrailBlock,
  notes: NotesBlock,
  document: DocumentBlock,
  table: TableBlock,
  card_collection: CardCollectionBlock,
  timeline: TimelineBlock,
  map: MapBlock,
};
```

Renderers receive validated data and invoke ordinary API commands for actions.
They do not receive credentials, provider clients, raw SQL, or executable code.
Each renderer must define loading, empty, stale, error, unsupported-version,
keyboard, and narrow-screen behavior.

## 9. Persistence and API shape

R5 should introduce a small durable model rather than storing the complete
Surface as opaque Event metadata:

- `surface_blocks`: Event ownership, type/version, title, posture, validated
  data, evidence references, creator/manager provenance, stable default layout
  hint, revision timestamps;
- `surface_preferences`: per-user Event layout mode, split ratio, positions,
  collapsed blocks, and selected block;
- an Event Surface revision source suitable for SSE invalidation.

Initial API:

```text
GET    /v1/events/{event_id}/surface
PUT    /v1/events/{event_id}/surface/blocks/{block_id}
DELETE /v1/events/{event_id}/surface/blocks/{block_id}
PATCH  /v1/events/{event_id}/surface/preferences
GET    /v1/surface/block-types
```

The existing `GET /events/{event_id}` remains compatible while the app moves
to the dedicated Surface document endpoint.

## 10. Acceptance gate

The typed Surface runtime is ready when:

1. Career OS can render a sourced company table and cited report without a
   Career-OS-specific React page;
2. a Trip fixture can render a map and itinerary from the same Surface API;
3. system Task/Decision/File/Trail blocks still read canonical records;
4. a model cannot inject HTML, script, CSS, unknown components, cross-Event
   record IDs, invalid URLs, or out-of-range coordinates;
5. live updates preserve selected block, scroll, and layout;
6. reconnect catches the latest Surface revision and terminal Task state;
7. unsupported block versions fail visibly and safely;
8. desktop split, conversation focus, Surface focus, and narrow-screen modes
   pass interaction and accessibility checks.
